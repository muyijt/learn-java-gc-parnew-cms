# learn-java-gc-parnew-cms
> 学习ParNew和CMS垃圾收集器，以及底层三色标记算法

## 分代收集理论
```text
新生代中每次收集有大量对象死去，所以选择“标记--复制算法”，老年代对象每次收集后对象对活率较高，所以选择“标记--清除算法”或“标记--整理算法”

注意：“标记--清除算法”或“标记--整理算法”要比“标记--复制算法”慢很多
```

## 垃圾收集算法

### 标记--复制算法
```text
将内存分为大小相同的两块，每次使用其中一块，当这块内存使用完后就将还存活的对象复制到另一块去，然后将已使用的那块内存的空间一次清理掉
```

### 标记--清除算法
```text
标记存活的对象，统一回收所有未被标记的对象
```

### 标记--整理算法
```text
标记存活的对象，让所有存活的对象向一端移动，然后直接清理掉端边界以外的内存
```

## Serial收集器
```text
单线程，只能用在年轻代，用复制算法。（几乎已不再使用）
-XX:+UseSerialGC  启用Serial垃圾收集器
```

## Serial old收集器
```text
单线程，只能用在老年代，用标记--整理算法。（几乎已不再使用）
-XX:+UseSerialOldGC  启用Serial Old垃圾收集器
```

## Parallel收集器
```text
多线程，只能用在年轻代，用复制算法
-XX:UseParallelGC  启用Parallel垃圾收集器
```

## Parallel old收集器
```text
多线程，只能用在老年代，用标记--整理算法
-XX:UseParallelOldGC  启用Parallel old垃圾收集器
```

## ParNew收集器
```text
多线程，只能用在年轻代，用复制算法
-XX:UseParNewGC  启用ParNew垃圾收集器
```

## CMS收集器
```text
多线程，只能用在老年代，用标记--清除算法

优点:
  并发收集，stw时间短。

缺点:
  1. 对cpu资源敏感（会和服务抢资源）
  2. 无法处理浮动垃圾（在并发标记和并发清理阶段又产生垃圾，这种浮动垃圾只能等到下次gc再清理）
  3. 使用标记--清除算法会导致收集结束后有大量空间碎片。可以通过参数-XX:+UseCMSCompactAtFullCollection可以让jvm在执行完标记清除后再做整理
  4. 执行过程中的不确定性，会存在上一次垃圾回收还没执行完，然后垃圾回收又被触发的情况，特别是在并发标记和并发清理阶段会出现，一边回收，系统一边执行，也许没回收完就再次触发full gc，也就是concurrent mode failure,此时会进入stop the world，并被替换为Serial old收集器来回收

核心参数:
  -XX:+UseConcMarkSweepGC  启用cms垃圾收集器
  -XX:+ConcGCThreads  并发的GC线程数（一般默认就可以，不建议改）
  -XX:+UseCMSCompactAtFullCollection  gc之后做压缩整理（减少空间碎片）
  -XX:CMSFullGCsBeforeCompaction  多少次gc之后压缩一次，默认是0，代表每次gc后都会压缩整理一次
  -XX:CMSInitiatingOccupancyFraction  当老年代使用达到该比例时会触发gc，默认是92，是百分比（比例设置的越小，老年代留下的空间就越多，出现concurrent mode failure概率就越小）
  -XX:+UseCMSInitiatingOccupancyOnly  只使用设定的回收阈值（-XX:CMSInitiatingOccupancyFraction设定的值），如果不指定，JVM仅在第一次gc使用设定值，后续则会自动调整
  -XX:+CMSScavengeBeforeRemark  在cms gc前启动一次minor gc，目的在于减少老年代对年轻代的引用，降低CMS GC的标记阶段时的开销，一般CMS的GC耗时80%都在标记阶段
  -XX:+CMSParallellnitialMarkEnabled  表示在初始标记的时候多线程执行，缩短stw
  -XX:+CMSParallelRemarkEnabled  在重新标记的时候多线程执行，缩短stw

工作步骤:
  1. 初始标记: 暂停工作线程(stw)，标记gc roots能直接引用的对象
  2. 并发标记: 从gc roots直接引用的对象开始遍历整个对象图的过程。这个过程耗时较长，但不需要停止工作线程。由于用户线程与gc线程共同执行，可能导致已标记过的对象的状态发生改变
  3. 重新标记: 暂停工作线程(stw)，修正并发标记阶段因为用户线程继续运行而导致的标记产生变动的那一部分对象的标记记录
  4. 并发清理: 开启用户线程，同时GC线程开始对未标记的区域做清扫。这个阶段如果有新增对象会被标记为黑色不做任何处理
  5. 并发重置: 重置本次GC过程中的标记数据

核心实现算法: 三色标记算法
  黑色: 表示对象已经被垃圾收集器访问过，且这个对象的所有引用也都扫描过
  灰色: 表示对象已经被垃圾收集器访问过，但这个对象上至少存在一个引用还没被扫描过
  白色: 表示对象尚未被垃圾收集器访问过。显然在可达性分析刚刚开始的阶段，所有的对象都是白色的，若在分析结束的阶段，仍然是白色的对象，即代表不可达
  
  多标--浮动垃圾
    在并发标记过程中，如果由于方法运行结束导致部分局部变量(gcroot)被销毁，这个gcroot引用的对象之前又被扫描过(被标记未非垃圾对象)，那么本轮gc不会回收这部分内存。这部分本应该回收但是没有会受到的内存，被称为浮动垃圾。
    浮动垃圾并不会影响垃圾回收的正确性，只是需要等到下一轮垃圾回收中才被清除。另外，针对并发标记和并发清理后产生的新对象，直接全部当成黑色，本轮不会惊醒清除
  
  漏标
    漏标会导致被引用的对象被当成垃圾误删除，这是严重bug。解决方案有“增量更新”和“原始快照”
    增量更新: 当黑色对象插入新的指向白色对象的引用关系时，将这个新插入的引用记录下来，等待并发扫描结束之后，再将这些记录过的引用关系中的黑色对象为根，重新扫描一次
    原始快照: 当灰色对象要删除指向白色对象的引用关系时，将这个要删除的引用记录下带，等并发扫描结束之后，再将这些记录过的引用关系中的灰色对象为跟，重新扫描一次，这样就能扫描到白色对象，并将白色对象直接标记为黑色
```

























> jdk8以及8之前的版本，推荐垃圾回收器组合为：年轻代用ParNew收集器，老年代用CMS收集器。
